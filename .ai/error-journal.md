# 错题本

> 每次修复成功后总结错误、抽象规则。
> 每次开始新改动前逐条检查自检清单。

## 错误 #1：切换语种后页面没有变化

**日期：** 2026-04-11
**现象：** 用户点击语种按钮后，URL 变了但页面没有真正切换到对应语种
**根因：** DJI Store 切换语种不仅需要修改 URL 路径段（如 `/cn/` → `/hk/`），还需要带上 `set_region` 参数（如 `set_region=HK`）。第一版脚本从 `hreflang` 标签推断数据，认为不需要 `set_region`，但实际弹窗链接里都带了这个参数
**你当时的思路：** 从页面 `<link rel="alternate" hreflang="...">` 标签提取路径，认为路径段足够，不需要额外参数
**为什么错了：** `hreflang` 标签是给搜索引擎用的 SEO 元数据，和实际用户跳转的链接参数不一定一致。应该以实际弹窗链接（用户交互的真实数据源）为准
**修复：** 爬取弹窗 DOM 中的 `a.switchcountry_click` 链接，提取真实的 `path` 和 `set_region` 参数，全部补回

### 抽象规则 R1：数据源必须来自用户交互的真实路径，而非 SEO/元数据标签
> 当需要模拟用户行为（如点击跳转）时，必须从用户实际交互的 DOM 元素（按钮、链接、表单）提取数据，而不是从 `<meta>`、`<link>`、`hreflang` 等元数据推断。元数据是给机器看的，和用户交互的行为可能不一致。

## 错误 #2：美国地区的 URL 路径被错误处理

**日期：** 2026-04-11
**现象：** 爬取弹窗数据时，United States 的 path 显示为 `product`，而非预期的国家代码
**根因：** DJI Store 美国版是默认地区，URL 没有语种前缀（`store.dji.com/product/...`），爬取脚本取 `pathname.split('/')[0]` 时拿到的是 `product` 而非国家代码
**你当时的思路：** 假设所有地区都有统一的路径前缀格式
**为什么错了：** 没有考虑"默认地区无前缀"这种特殊情况。很多多语言网站的默认语言版本 URL 是不带前缀的
**修复：** 为美国设置 `path: null`，切换逻辑中用 `knownPageSegments` 列表判断是否有语种前缀，`null` 时移除前缀

### 抽象规则 R2：多语言 URL 切换必须处理"默认语言无前缀"的特殊情况
> 多语言网站通常有一个默认语言版本不带路径前缀。设计 URL 切换逻辑时，必须区分"有前缀"和"无前缀"两种状态，不能假设所有语种都有统一格式。

## 错误 #3：`overflow` 裁掉了超出容器边界的子元素

**日期：** 2026-04-11
**现象：** 面板左侧的收起/展开按钮（`position: absolute; left: -22px`）不可见
**根因：** 面板设置了 `overflow-y: auto` 用于内容滚动，但 `overflow` 会同时裁掉 X 轴上超出容器边界的绝对定位子元素
**你当时的思路：** `overflow-y: auto` 只影响 Y 轴滚动，X 轴超出的元素应该正常显示
**为什么错了：** CSS 规范中，`overflow-x` 和 `overflow-y` 是联动的——当其中一个设为非 `visible` 时，另一个也会自动变成非 `visible`（通常变为 `auto`），导致两个方向都会裁剪
**修复：** 将收起按钮改为独立的 `fixed` 定位元素，脱离面板 DOM 层级，不受面板 `overflow` 影响

### 抽象规则 R3：需要超出容器边界显示的元素，不能作为 `overflow` 容器的子元素
> 当容器需要滚动（`overflow: auto/scroll`）时，任何需要在容器边界外显示的元素必须放到容器外部，使用独立的定位方式。CSS `overflow` 在任一方向设为非 `visible` 时，另一方向也会被隐式裁剪。

## 错误 #4：新增页面段（`edit`）未加入已知路径段列表，导致路径被错误替换

**日期：** 2026-04-11
**现象：** 在 `dbeta.me/edit/handheld/...` 页面切换语种时，`edit` 被当作语种前缀替换掉了，变成 `dbeta.me/cn/handheld/...`（`edit` 消失）
**根因：** `knownPageSegments` 列表中只有 `mobile`、`product` 等，没有 `edit`。脚本用这个列表判断 URL 第一段是否为语种前缀，`edit` 不在列表中被误判为语种前缀
**你当时的思路：** 只考虑了已知的 URL 格式，没有预见到新的路径段出现
**为什么错了：** 采用"白名单判断非语种路径段"的方式，任何新增的页面段都必须手动加入白名单，否则会被误判。这种方式对变化不友好
**修复：** 将 `edit` 加入 `knownPageSegments` 列表

### 抽象规则 R4：白名单式判断必须考虑新增项的维护成本，并在添加时覆盖所有使用该白名单的分支
> 当使用白名单（已知值列表）做逻辑判断时，每次系统新增路径/类型/状态，都必须检查并更新所有使用该白名单的地方。如果白名单分散在多处，应统一到一个配置中。

## 错误 #5：版本号未升级导致 Tampermonkey 认为"没有更新"

**日期：** 2026-04-11
**现象：** 脚本内容已更新并同步到 Gist，但用户点击"检查更新"后提示"暂时没有更新"
**根因：** 修改了脚本内容但忘记递增 `@version`，Tampermonkey 通过版本号比较判断是否有新版本，版本号相同时不会触发更新
**你当时的思路：** 认为内容变了就会被检测到更新
**为什么错了：** Tampermonkey 的更新检测基于版本号语义比较，不是内容 diff
**修复：** 每次推送修改时同步递增版本号

### 抽象规则 R5：所有依赖版本号做更新检测的系统，每次内容变更必须同步递增版本号
> 当分发系统（包管理器、插件市场、油猴脚本）通过版本号判断是否有更新时，任何内容变更都必须伴随版本号递增，否则更新不会被检测到。这应该作为发布流程的强制步骤。

## 自检清单（每次改动前必读）

- [ ] R1: 我使用的数据来源是用户实际交互的 DOM 元素，而非 SEO 元数据或推断？
- [ ] R2: URL 切换逻辑是否处理了"默认语言无路径前缀"的情况？
- [ ] R3: 需要在容器外显示的元素，是否放在了 `overflow` 容器的外部？
- [ ] R4: 新增了路径段/类型/状态时，是否更新了所有相关的白名单？
- [ ] R5: 内容有变更时，版本号是否已递增？
